Skyward Platform — App Hierarchy

client → property → project. The conceptual model and how it plays out in three scenarios.

The model

Client
Commercial entity
The legal counterparty who signs the SOW. Phil Lasry LLC, TNA Pty Ltd, Tacoma Legacy Partners LLC.
Property · has 1+
Operational unit
Brand + website + pipeline state. Owns Brand DNA, pages, audits. One client can have many. (primary_domain, url_prefix) uniquely identifies.
Project · has 1+
Playbook execution
A channel-scoped run. type ∈ { seo, paid_search, paid_social }. Project type determines which tabs render on the property page.
Cross-relationships: client ↔ client via related_clients[] (soft affinity) property ↔ property via parent_property_id (Brand DNA inheritance) SOW funds projects (lives in Scope Builder)

Two distinct clients (separate SOW signers) sharing one domain. KS Provo's site lives at kitchenguard.com/provo. Brand DNA inherits from KG Corporate via parent_property_id — crosses the client boundary via the related_clients[] affinity. Different project types lighting up different UI on each property page (SEO tabs on KG root, Campaigns tabs on KS Provo).

What lives where (interim, 2026-05-14):  Client + property + brand DNA + pages → Supabase (this app owns).  Project list + project↔domain mapping → BigQuery Meta (Adam owns; surfaced via skyward-common).  SOW + line items → Scope Builder (separate SQLite, future read-only sync).
